# Go Development

## Directory Layering

The Go codebase lives under `internal/` and follows clean architecture (ports-and-adapters):

```
internal/
  api/v1/              HTTP routing, request/response, auth middleware
    repository/        API modules (one per domain, registers routes)
  repository/          Domain types, interfaces, business logic
    dal/               Store/Authorizer implementations backed by sqlc
  dal/                 Data Access Layer
    queries/           Source SQL files for sqlc (edit these)
    daldb/             sqlc-generated Go code (never hand-edit)
    schema/sqlite/     SQLite schema DDL and seed data
    metastore/         OCI MetadataStore implementation
    dbcore/            Database lifecycle (bootstrap, migrations)
  oci/                 OCI/registry domain types and interfaces
    storage/           OCI storage abstraction and drivers
  registry/
    distribution/      Distribution library bridge and middleware
  config/              Configuration loading and validation
  server/              HTTP server lifecycle and TLS
  cmd/                 CLI commands (serve, install, migrate, config)
  auth/                Authentication primitives (BasicAuthenticator)
  logging/             Structured logging (slog-based)
  certs/               Certificate management
  system/              Runtime utilities (env, fs, health)
```

Dependencies flow inward: `api/v1/` depends on domain interfaces in `repository/` and `oci/`, never on DAL implementations directly. The `cmd/serve.go` file wires everything together.

## sqlc Workflow

Database queries use [sqlc](https://sqlc.dev/) for type-safe code generation. Config is in `sqlc.yaml` at the repo root.

**Source files** (edit these):
- `internal/dal/queries/*.sql` — annotated SQL queries with `-- name:` directives
- `internal/dal/schema/sqlite/quay_schema.sql` — schema DDL

**Generated files** (never hand-edit):
- `internal/dal/daldb/*.sql.go` — generated Go query functions
- `internal/dal/daldb/db.go` — `DBTX` interface and `Queries` struct
- `internal/dal/daldb/models.go` — table model structs

All generated files carry the header `// Code generated by sqlc. DO NOT EDIT.`

**Adding or modifying a query:**

1. Edit the `.sql` file in `internal/dal/queries/` using sqlc annotation syntax:
   ```sql
   -- name: GetWidgetByName :one
   SELECT id, name FROM widget WHERE name = ?;
   ```
2. Run `sqlc generate` from the repo root
3. Use the generated function in your DAL implementation
4. Commit both the `.sql` source and the regenerated `.sql.go` files

**Never hand-edit files in `internal/dal/daldb/`.** If a query needs changing, modify the source `.sql` file and regenerate.

## Service Layer Patterns

### Interfaces

Domain packages define interfaces; DAL packages implement them:

```go
// internal/repository/service.go
type Store interface {
    Get(ctx context.Context, ref Ref) (Repository, error)
    SetVisibility(ctx context.Context, id int64, visibility Visibility) error
    MarkDeleted(ctx context.Context, repo Repository, deletedName string) error
}

type Authorizer interface {
    CanAdminRepository(ctx context.Context, principal *auth.Principal, repo Repository) (bool, error)
}
```

DAL implementations use compile-time interface checks:

```go
// internal/repository/dal/store.go
var _ repository.Store = (*Store)(nil)
```

### Constructor Validation

Constructors validate required dependencies and return errors for nil arguments:

```go
func NewService(store Store, authorizer Authorizer) (*Service, error) {
    if store == nil {
        return nil, fmt.Errorf("nil repository store")
    }
    if authorizer == nil {
        return nil, fmt.Errorf("nil repository authorizer")
    }
    return &Service{store: store, authorizer: authorizer}, nil
}
```

### Sentinel Errors

Domain packages define sentinel errors in `errors.go`:

```go
// internal/repository/errors.go
var (
    ErrNotFound          = errors.New("repository not found")
    ErrForbidden         = errors.New("forbidden")
    ErrInvalidVisibility = errors.New("invalid visibility")
)
```

Check errors with `errors.Is()`, not string comparison.

## API Modules

HTTP endpoints are organized as modules that register with the router:

```go
// internal/api/v1/api.go
type Module interface {
    Register(*Router)
}
```

Each domain has a module package (e.g., `internal/api/v1/repository/`) that implements `Module` and registers its routes. Modules depend on domain service interfaces, not DAL implementations.

Authentication uses the `Authenticator` interface and the `RequireBasic()` middleware for endpoints that need auth.

## Testing

### Go Test Commands

```bash
# Run all Go tests
go test ./internal/...

# Run tests for a specific package
go test ./internal/repository/... -v

# Run a single test
go test ./internal/repository/... -run TestServiceDelete -v
```

### Test Patterns

**Unit tests with fakes** (`internal/repository/service_test.go`):
- In-package fakes implement domain interfaces (`fakeStore`, `fakeAuthorizer`)
- One test function per scenario (not table-driven for service tests)
- Use `t.Fatalf()` for assertions, not testify
- Use `t.Context()` for context (Go 1.24+)

**Table-driven subtests** (`internal/repository/dal/store_test.go`):

```go
for _, tc := range []struct {
    name    string
    result  sql.Result
    wantErr error
}{
    {name: "driver error", ...},
    {name: "no rows", ...},
} {
    t.Run(tc.name, func(t *testing.T) {
        // ...
    })
}
```

**Integration tests with SQLite** (`internal/api/v1/repository/visibility_test.go`):
- `setupAPITestDB(t)` creates an in-memory SQLite database with full schema
- Seeds roles, visibility levels, users, and repositories
- Tests exercise the full HTTP handler stack

**MetadataStore tests** (`internal/dal/metastore/metastore_sqlite_test.go`):
- `setupStore(t)` uses `dbcore.Setup()` with the real embedded schema
- Tests run against the production schema and seed data

### Test Conventions

- Place test files next to the code they test (`*_test.go` in the same package)
- Use the standard `testing` package; testify is available but not used in existing tests
- Use `t.Context()` instead of `context.Background()`
- Name test functions `TestServiceChangeVisibility`, not `Test_service_change_visibility`

## Linting

The repo uses golangci-lint with 20+ linters (config in `.golangci.yml`):

```bash
# Run linter
golangci-lint run ./internal/...

# Run with auto-fix
golangci-lint run ./internal/... --fix
```

Formatters: `gofmt` and `goimports` are enabled via golangci-lint.
